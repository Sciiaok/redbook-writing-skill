不是。更准确的结论是：这两条都是被大量转载的运营口诀，不能当成小红书已证实的当前算法。

| 说法 | 证据状态 | 结论 |
|---|---|---|
| CES＝点赞×1＋收藏×1＋评论×4＋分享×4＋关注×8 | `D / 未证实`：没有可追溯的一手公式，转载文章也没有分母、模块、时期或失败样本 | 不用于打分、选题或 KPI |
| 每篇笔记先给 200 流量，再逐级放量 | `D / 未证实`：没有一手材料给出固定首轮数量或保证曝光 | 新内容冷启动存在，但不等于固定 200 流量池 |

截至 2026-07-16，可核的一手材料只支持这些有限结论：

- [国家网信办的小红书个性化推荐算法备案](https://beian.cac.gov.cn/api/static/fileUpload/principalOrithm/additional/user_c015445c-80ac-45f7-94d7-3871e961b1fe_d4425f3b-7f35-45af-b8d4-badd4424d6d5.pdf)说明笔记内容数据和用户交互行为会参与推荐，并列出关注、点击、搜索、点赞、收藏、分享、评论等行为；它没有公布这些行为的相对权重。
- 小红书团队 2026 年的 [IDProxy 冷启动论文](https://arxiv.org/html/2603.01590v1)确认“新发布内容缺少历史行为”这个冷启动问题真实存在，并披露了用图文语义代理接入 CTR 排序、同时观察阅读、时长和互动；论文没有披露“先给 200 曝光”或固定分级流量池。

还有一个逻辑问题：“200 流量”没有定义分母。曝光次数、点击/阅读数（小眼睛）、独立读者数和推荐页触达不是同一个指标，不能混用。

## 为什么我不直接编一张“新号前 7 天标准表”

7 天可以是你的项目周期，但不是已证实的平台周期。你还没有给出类目、人群、业务目标、素材库存、单篇制作成本、审核时延和后台基线；此时写“每天几篇、固定比例、标题字数、24/72 小时及格线”，数字只能来自想象，不是生产级 SOP。

| 你要的数字 | 应由什么推导 |
|---|---|
| 每天发几篇 | 当天通过质量门的成稿库存、可用生产/审核时长、单篇合格成本 |
| 内容比例 | 业务目标、每篇的首要任务、证据充足且可生产的选题库存 |
| 标题字数 | 查询/推荐任务、标题预览是否保留核心承诺、可读性；不是算法权重 |
| 24/72 小时达标线 | 同账号、同任务、同内容年龄的后台基线与分布；新号尚无该基线 |

因此，我不会把这些缺失依赖重新包装成 Day 1—Day 7 日历。

## 可立即执行的替代 SOP

```text
定位与证据门
  → 生产一篇合格内容
  → 发布并记录同口径漏斗
  → 等到主指标有可解释事件量
  → 只改一个变量再测试
  → 用本账号数据更新基线
```

1. 每篇发布前只选一个首要任务：`推荐触达 / 搜索承接 / 关系建立 / 商业转化`。对应记录曝光→点击→消费深度→互动→主页/关注→合规后链路，不能拿点赞代替所有结果。
2. 发布门是：有可追溯需求证据、只服务一个人群的一个具体处境、素材真实可生产、标题/封面/正文兑现同一承诺、合规审查通过。当天发布量＝当天同时满足发布门和真实产能的笔记数；不预设通用篇数。
3. 内容比例不先拍脑袋。候选内容按“业务优先级、证据就绪、素材就绪”排队；实际比例＝各首要任务的合格发布数÷全部合格发布数。只有本账号结果出现后才调整。
4. 标题不设通用字数。通过标准是：目标用户无需展开正文，就能识别“这是不是我的处境、能得到什么答案”；核心承诺不能依赖被截断部分，删掉不改变预期的词。
5. 如果团队必须在 24/72 小时看数据，可以把它们设为**记录点，不是达标线**。新号首批笔记状态统一记为 `baseline_building`。有了可比较样本后，再计算：

```text
同龄相对表现 ＝ 本篇在该记录点的主指标
              ÷ 同任务、同类题材、相近账号阶段笔记的主指标中位数
```

结果只允许写 `win / loss / inconclusive`。事件不足、方向不稳定或混杂太多时必须写 `inconclusive`，不为了复盘强行判输赢。热点内容和长期搜索内容的观察窗口也应分别按生命周期确定，不能统一锁死在 24/72 小时。

要把它进一步推导成真实的每日排期，请补齐这一行：

```text
类目＝｜目标用户＋具体处境＝｜首要业务目标＝｜每日可用生产/审核时长＝
合格单篇耗时＝｜现有真实素材＝｜审核时延＝｜后台可见字段＝｜已有同口径笔记＝
```

若“已有同口径笔记＝0”，首个周期的目标就是建立基线，而不是用网上统一阈值判定账号成败。
