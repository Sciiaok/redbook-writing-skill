不能把这套说法作为事实写进 Skill。

- `CES=点赞×1、收藏×1、评论×4、转发×4、关注×8`：未验证，流传版本互相冲突。
- “达到固定 200 分进入下一级流量池”：未验证，没有找到对应的一手平台依据。
- 现有公开资料只能确认内容与互动行为会参与推荐，不能证明固定权重、固定曝光池或统一晋级线。

Skill 中可以收录为一条“待审计传言”：

```yaml
claim: CES 精确权重与固定 200 分晋级
status: unknown
evidence_grade: D
allowed_action:
  - 记录为传言
  - 检查原始来源、分母、入口、时期和反例
  - 用本账号漏斗数据设计准实验
forbidden_action:
  - 按公式给笔记打分
  - 把 200 写成平台晋级线
  - 据此制定统一首周发布量或互动指标
```

新号 SOP 应改成依赖关系，而不是编造日历：

1. 先确认真实产能、内容审核时延、可用素材和首要任务。
2. 发布达到质量门的内容，积累同口径自然基线。
3. 每篇只设一个首要任务，并记录曝光、点击、消费深度、互动、主页访问或关注中实际可取得的字段。
4. 等主指标达到可解释事件量；最低事件量必须由账号基线和指标分母推导，不能套固定时点。
5. 每轮只改变标题、封面、载体、结构或发布时间中的一个变量。
6. 用多篇内容的中位数和分布比较；事件不足时结论写 `inconclusive`。
7. 根据失败层级调整：曝光不足查需求与分发入口，点击不足查封面标题，消费不足查正文兑现，关注不足查主页承诺。

要生成真正带数字的排期，还缺四类输入：单篇制作成本、可持续产能、账号后台基线、各内容任务可取得的指标。没有这些数据时，任何“每天发几篇、目标多少互动、固定短窗与长窗阈值”都只是另一套未经验证的玄学。
