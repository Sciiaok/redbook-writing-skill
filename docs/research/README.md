# 研究资产索引

这个目录是 `redbook-writing` 的来源室。README 里出现的“官方课”“操盘复盘”“站内高低对照”，都应该能从这里找到原始审计笔记和机器副本。

资料入库时保留四件事：它是什么来源，发布者有什么激励，原文能支持什么，不能支持什么。同一句经验在三门课里出现，不会因此升级成平台因果；公开赞藏评也不会改名为自然曝光。

## 为什么研究 JD、brief 和其他“不像运营课”的材料

公开运营课通常告诉你“可以怎么做”。真正生产时还缺许多变量：谁提供素材，哪个版本能发，什么结果要复核，哪些说法需要授权，付费投放和自然结果如何分账。这些往往出现在旁边的文件里。

| 材料 | 重点抽取的生产字段 | 进入 Skill 后的用法 |
| --- | --- | --- |
| JD / 岗位访谈 | 交付物、角色分工、复核点、返工原因 | 设计 handoff 和 review 合同，不把“会做”当流量证明 |
| 创作者 brief / 合作规范 | 必需素材、禁用说法、披露、版本、审批 | 变成 asset manifest、授权和商业门禁 |
| 品牌/代理结案 | 选品、人群、创意、投放、搜索、复盘的混合变量 | 抽生产动作，同时标记无法归因给单个标题/封面的部分 |
| 采购、供给和归因文件 | 数据口径、供给来源、版本血缘、交付验收 | 防止不同 surface、不同素材和不同分母被混成一个“增长” |
| 学术论文/公开数据 | 样本、变量、方法、显著性、偏差 | 把“相关”和“可生产的因果”分开，定义后续实验 |
| 站内高/普通/低帖 | 同系列常量、真正差异、反例和素材限制 | 先形成 `task_fit` 或待验证假设，不把单篇高赞造型做模板 |

## 从资料到成稿，中间经过什么

```text
公开课程 / 项目复盘 / 隐性生产物 / 学术研究 / 站内样本
  → 专题审计（can support / cannot support）
  → 带 source_refs 的候选机制卡
  → content + carrier/truth + learning/governance 三槽检索
  → 真实素材 manifest + 视觉方向骨架
  → SQLite 中 exact category/job/carrier 的 published style binding
  → draft binding + rendered assets + validator
  → 发布后的一方 exposure / job outcome / diagnostics
```

长文研究不会在每次成稿时全量塞进上下文。归一化资产在 [`traffic-mechanisms-v1.json`](../../redbook-writing/assets/traffic-mechanisms-v1.json)，人工可读解释在 [`traffic-mechanism-library.md`](../../redbook-writing/references/traffic-mechanism-library.md)，视觉骨架在 [`visual-direction-cards-v1.json`](../../redbook-writing/assets/visual-direction-cards-v1.json)。类目自有的逐页观察、原图和 SQLite 保持本地私有；仓库只提交脱敏观察、hash、方法与反例，不把第三方原图或完整正文收进 Git。

晋级路径固定为：

```text
lead
  → production_hypothesis 或 task_fit
  → 合格高低对照下的 public_proxy_association
  → 本账号一方 exposure 复现后的 first_party_traffic_validated
```

当前最后一级为 0。[`qualified-cells.json`](qualified-cells.json) 同时记录了风格格子为什么还不合格：performance receipt 不可复算、逐页素材 hash 不完整、权利状态未验证、matched control 不足。这些缺口闭合前，starter pack 保持禁用。

## 怎样自己核对

在仓库根目录执行：

```bash
# 机制数、引用次数和去重 ref
jq '{mechanisms:(.mechanisms|length),
     citations:([.mechanisms[].source_refs[]]|length),
     unique_refs:([.mechanisms[].source_refs[].ref]|unique|length)}' \
  redbook-writing/assets/traffic-mechanisms-v1.json

# 查某张卡的证据范围、动作和失效条件
jq '.mechanisms[] | select(.mechanism_id=="TM19") |
    {name,mechanism_kind,evidence,actions,failure_conditions,source_refs}' \
  redbook-writing/assets/traffic-mechanisms-v1.json

# 视觉卡数与候选状态
jq '{cards:(.cards|length), self_check}' \
  redbook-writing/assets/visual-direction-cards-v1.json

# 当前风格格子发布门
jq '{status,qualified_cell_count,starter_pack_action,gap_reasons}' \
  docs/research/qualified-cells.json
```

想审计基础快照里的某个数字，从 [`evidence-snapshots/2026-07-16-platform-mechanisms/`](../../evidence-snapshots/2026-07-16-platform-mechanisms/) 的 `query-log.csv → source-log.csv → claim-ledger.csv` 往回查。这比相信一句“我们调研了很多资料”更有用。
